# V2 inventory — size + bandwidth report

Generated by `steplibrary/specgen/cmd/steplib-gen` against the real `bitrise-steplib` repo at
commit `b9af7d7` (444 active steps, 3565 versions).

This report is the PoC A deliverable that answers two questions for team
review:

1. Does the V2 layout meaningfully reduce per-build client bandwidth vs V1?
2. What does it cost in terms of file count / total inventory size at the origin?

## TL;DR

| Scenario | V1 today | V2 |
|---|---:|---:|
| Origin total size (raw) | ~24 MB (single file) | 24.31 MB (5,236 files) |
| Origin total size (gzipped) | ~5.7 MB | **7.42 MB** |
| Per-workflow bandwidth — fixed versions (10 steps) | ~5,700 KB | **12.8 KB** |
| Per-workflow bandwidth — `latest` resolution (10 steps) | ~5,700 KB | **17.3 KB** |
| Per-workflow bandwidth — minor-locked (10 steps) | ~5,700 KB | **19.3 KB** |
| One-shot catalog (WFE / Integrations Page) | ~2.2 MB (slim-spec.json) | **46.6 KB** |

Per-workflow bandwidth drops by a factor of **~330×** for fixed-version
workflows and **~290×** even for the most expensive constraint
(minor-locked, requiring per-step `versions.json` fetches). The catalog
case (WFE / Integrations Page) drops by **~47×** vs today's `slim-spec.json`.

The origin total is slightly larger gzipped than V1's single `spec.json`
because (a) each file gzips less efficiently in isolation than one big
file, and (b) we now host per-step `step-info.json`, `latest.json`,
`versions.json`, and pre-resolved catalog data. The tradeoff is heavily in
favor of clients: origin pays once, every build benefits.

## Per-workflow bandwidth (client side, gzipped)

Modeled on a workflow with 10 distinct steps, fresh cache (worst case).
Bytes assume HTTP/1.1 + gzipped responses, ignoring TLS handshake
overhead (irrelevant given the small payloads and modern keep-alive).

### V1 baseline

In V1, every build that triggers a steplib update fetches the entire
`spec.json`:

```
spec.json (S3 HEAD): 24,431,255 bytes raw, ~5.7 MB gzipped
```

After parsing, all 5.7 MB is in memory regardless of which 10 steps the
workflow uses.

### V2 — fixed version pins (`step@1.2.3`)

Workflow uses only fixed semver pins. No `latest.json` or `versions.json`
needed.

```
10 × step.json   = 10 × 1,277  = 12,770 bytes ≈ 12.5 KB
```

Reduction vs V1: **~445× smaller**.

### V2 — `latest` resolution

Workflow uses `latest` (or unspecified version). For each step, fetch
`spec/steps/<id>/latest.json` to resolve, then `step.json`.

```
spec/step_ids.json       = 3,487 bytes    (one-time, validates IDs)
10 × latest.json         = 10 × 101 = 1,010 bytes
10 × step.json           = 10 × 1,277 = 12,770 bytes
─────────────────────────────────────────
TOTAL                    = 17,267 bytes ≈ 16.9 KB
```

Reduction vs V1: **~330× smaller**.

### V2 — `MajorLocked` (`step@1.x.x`)

Same as `latest` resolution above — `latest.json` already carries
`latest_by_major`, so no extra round trip.

### V2 — `MinorLocked` (`step@1.2.x`)

Falls through to per-step `versions.json` for resolution.

```
spec/step_ids.json       = 3,487 bytes
10 × versions.json       = 10 × 309 = 3,090 bytes
10 × step.json           = 10 × 1,277 = 12,770 bytes
─────────────────────────────────────────
TOTAL                    = 19,347 bytes ≈ 18.9 KB
```

Reduction vs V1: **~294× smaller**.

### V2 — Catalog browse (WFE / Integrations Page)

Single fetch of the fat catalog gives every consumer the info needed for a
browse view across all 444 steps.

```
spec/latest_versions.json = 46,551 bytes ≈ 45.5 KB
```

Reduction vs today's `slim-spec.json` (2.2 MB gzipped on S3): **~47× smaller**.

When a user clicks into a specific step in WFE, that triggers one
`step.json` fetch (~1.3 KB gzipped). The flow scales naturally with
engagement instead of paying the catalog cost on every hit.

## What about cached / repeat builds?

V1 today re-checks `spec.json` ETag on every "steplib update" (which is
triggered by `latest` or `MajorLocked`/`MinorLocked` constraints). Even a
304 still costs the HTTP round trip and TLS, and any real change forces a
full re-download.

V2 lets every file revalidate independently:

- `step_ids.json` and `latest.json` revalidate cheaply (small files, fast 304).
- `step.json` files are **immutable** — once cached, no revalidation needed for a year. A repeat build with the same fixed-version pins pays **zero** bytes after first cache.
- Only the per-step `versions.json` and the catalog mutate frequently; both are sub-KB to KB-scale.

So the real-world distribution of build bandwidth is even more skewed in
V2's favor than the cold-cache numbers above suggest.

## Origin tradeoffs

V2 inflates the origin somewhat:

- **File count: 1 → 5,237.** A non-issue for object storage (S3 / GCS /
  R2 all handle millions of objects per bucket without complaint), but
  the release flow needs to upload many small files instead of one big
  one. Concurrent uploads (default `gsutil` / `aws s3 sync`) handle this
  in seconds.
- **Gzipped total: ~5.7 MB → 7.42 MB (+30 %).** Each file gzips less
  efficiently in isolation than one large file; we also pay for new index
  files that V1 doesn't have (per-step latest/versions/info JSONs). This
  cost is fixed at the origin; clients see massive savings on every
  build, so the trade is heavily in favor of the system as a whole.
- **Asset copy: 2.3 MB.** V2 brings step icons / screenshots into the
  inventory tree at `steps/<id>/assets/`. The catalog references them
  via inventory-root-relative paths (`"steps/<id>/assets/icon.svg"`),
  so the V2 inventory is self-sufficient and portable across hosting
  changes. The V1 parallel S3 mirror can keep existing for V1
  consumers; V2 no longer references it.

## Reproducing this report

From the stepman repo root, with a local clone of `bitrise-steplib`
checked out somewhere:

```sh
go run ./steplibrary/specgen/cmd/steplib-gen \
  -input /path/to/bitrise-steplib \
  -output /tmp/v2-out \
  -commit-sha "$(git -C /path/to/bitrise-steplib rev-parse HEAD)"
```

The exact numbers in this report were captured at commit `b9af7d7` of the
bitrise-steplib repo. Different commits will produce slightly different
counts as new steps and versions are released; the per-file medians and
the V1 / V2 ratio are stable.
